# Hito 0 — Preparación de base para arrancar

**Objetivo:** dejar listo todo lo necesario para empezar desde cero sin deuda inicial.

**Contexto ampliado para agentes IA**
Este hito no construye todavía funcionalidad de negocio, pero es crítico porque fija el terreno donde se apoyará todo lo demás. El documento deja claro que Zenoytdl no es un script puntual, sino una herramienta con arquitectura propia, capas bien separadas y proyección a futuro. Eso significa que empezar “rápido” pero sin estructura sería contraproducente: rompería la coherencia del modelo, haría más difícil la validación y comprometería la estabilidad del core. Aquí se prepara el repo, las convenciones, el entorno y la base mínima para que los siguientes hitos se implementen sin improvisación.

**Prerrequisitos:** ninguno.

**Checklist:**

* Definir stack técnico inicial del core.
* Crear estructura de repo y módulos base.
* Establecer convención de nombres, layout y separación por capas.
* Preparar entorno de desarrollo, linting, testing y empaquetado.
* Definir estrategia de configuración por YAML.
* Fijar política de logs, errores y trazabilidad mínima.
* Preparar entorno local con ytdl-sub, ffmpeg y SQLite disponibles.
* Documentar supuestos iniciales y límites del alcance.
* Dejar claro que TUI y GUI web quedan fuera del proyecto actual.
* Crear fixtures y ejemplos mínimos de configuración para desarrollo.

**Plan de pruebas:**

**Pruebas de sistema**

* Verificar que el proyecto arranca en entorno limpio.
* Verificar que dependencias críticas están accesibles.
* Verificar que la estructura base del repo coincide con la arquitectura prevista.

**Unitarias**

* Tests de utilidades base de paths, logging y carga de entorno.
* Tests de helpers de configuración global mínima.

**Integración**

* Verificar integración básica con SQLite.
* Verificar detección de ytdl-sub y ffmpeg desde el runtime.

**Funcionales**

* Confirmar que el comando o entrypoint principal responde.
* Confirmar que se puede cargar configuración mínima sin ejecutar lógica real.

**E2E**

* Flujo mínimo: arrancar proyecto, cargar config dummy, salir sin error.

**Prueba de regresión**

* Ejecutar bootstrap completo en entorno limpio y confirmar:

  * estructura correcta,
  * dependencias disponibles,
  * test runner operativo,
  * entrypoint ejecutable.

# Hito 1 — Preparación de contenedorización y runtime objetivo

**Objetivo:** dejar decididos desde el principio los condicionantes técnicos necesarios para que el core pueda terminar como herramienta desplegable en contenedor Docker, sin contaminar todavía la implementación del núcleo.

**Contexto ampliado para agentes IA**
Este hito existe para evitar un error clásico: desarrollar todo el core pensando implícitamente en un entorno local arbitrario y solo al final intentar “meterlo en Docker”. En este proyecto eso sería especialmente mala idea porque el documento ya presupone rutas persistentes, SQLite, caché, artefactos compilados y ejecución por subprocess sobre ytdl-sub. Por tanto, aquí no se dockeriza todavía, pero sí se fijan las reglas del juego del runtime objetivo: filesystem esperado, resolución de binarios, política de volúmenes, logs y persistencia. Eso permite que todos los hitos posteriores nazcan con compatibilidad container-first, aunque el empaquetado real llegue al final.

**Prerrequisitos:** hito 0.

**Checklist:**

* Definir oficialmente que el runtime objetivo soportado será Linux containerizado.
* Fijar la estrategia de imagen base.
* Fijar cómo se resolverán dentro del contenedor `ffmpeg`, `ffprobe` y `ytdl-sub`.
* Definir layout de directorios persistentes dentro del contenedor:

  * `/data/library`
  * `/data/tmp`
  * `/data/logs`
  * `/data/state.sqlite`
  * `/data/cache.sqlite`
  * `/data/compiled-ytdl-sub`
* Definir política de volúmenes y qué datos deben sobrevivir al reinicio del contenedor.
* Definir estrategia de configuración:

  * bind mount de YAMLs
  * variables de entorno
  * combinación de ambas
* Definir usuario/grupo de ejecución dentro del contenedor.
* Definir política de logs para contenedor:

  * stdout/stderr
  * logs JSON opcionales en volumen
* Definir si habrá un solo proceso principal o supervisor mínimo.
* Definir criterio de healthcheck de contenedor.
* Definir cómo se validará el contenedor en CI/CD o en pipeline local.

**Plan de pruebas:**

**Pruebas de sistema**

* Verificar que todas las decisiones de runtime containerizado son coherentes con el diseño del core.

**Unitarias**

* Tests de helpers de resolución de rutas Linux/container.
* Tests de generación o resolución de paths persistentes.

**Integración**

* Validar que la estrategia de volúmenes encaja con SQLite, caché, compilados y logs.

**Funcionales**

* Comprobar que la configuración puede expresarse sin ambigüedades para entorno local y entorno container.

**E2E**

* Simulación de arranque en entorno pseudo-container con layout `/data/...` y resolución de binarios prevista.

**Prueba de regresión**

* Ejecutar una batería de verificación de runtime objetivo y asegurar que no se rompen:

  * rutas base,
  * resolución de binarios,
  * política de persistencia,
  * contrato de logs para contenedor.

# Hito 2 — Modelado del dominio interno

**Objetivo:** definir el lenguaje interno real de Zenoytdl.

**Contexto ampliado para agentes IA**
Este es el primer hito verdaderamente “de producto”. Aquí se decide cómo piensa Zenoytdl el problema. El documento insiste en que el valor principal de la herramienta está en ofrecer una semántica más clara que ytdl-sub: perfiles, suscripciones, postprocesados, overrides, estado, caché, colas y control de ejecución. Por eso el dominio no puede ser una traducción pobre de ytdl-sub, sino un modelo propio, estable y expresivo. Si este hito queda bien hecho, los siguientes módulos se apoyarán sobre conceptos nítidos; si queda mal hecho, toda la herramienta se volverá ambigua.

**Prerrequisitos:** hito 1.

**Checklist:**

* Definir entidades principales: general config, perfil, suscripción, postprocesado, override, configuración efectiva, artefacto compilado, trabajo, estado.
* Definir enums y tipos base.
* Modelar relaciones perfil → suscripción → configuración efectiva.
* Fijar invariantes del dominio.
* Diferenciar modelo Zenoytdl y modelo ytdl-sub.
* Dejar el dominio desacoplado de API, cola y persistencia concreta.

**Plan de pruebas:**

**Pruebas de sistema**

* Verificar que el dominio puede inicializarse completo sin infraestructura adicional.

**Unitarias**

* Construcción válida de cada entidad.
* Validación de invariantes del dominio.
* Igualdad, serialización interna y normalización de tipos.

**Integración**

* Verificar que distintos objetos del dominio se relacionan correctamente.
* Verificar que el dominio no depende de infraestructura concreta.

**Funcionales**

* Crear perfiles, suscripciones y postprocesados en memoria.
* Resolver asociaciones básicas entre entidades.

**E2E**

* Flujo corto: crear perfil + suscripción + override y obtener estructura interna coherente.

**Prueba de regresión**

* Cargar un conjunto canónico de objetos de dominio y comprobar que:

  * las entidades existen,
  * las relaciones son válidas,
  * no hay acoplamiento accidental con ytdl-sub, DB o API.

* Redactar ejemplos de referencia válidos.
* Garantizar semántica de alto nivel y no detalle crudo de ytdl-sub.

**Plan de pruebas:**

**Pruebas de sistema**

**Integración**

## Regla operativa de regresión acumulativa

Se mantiene exactamente la misma idea que ya habías aceptado, ahora con numeración secuencial completa:

* al terminar el **Hito 0**, se ejecuta regresión del **Hito 0**
* al terminar el **Hito 1**, se ejecutan regresiones de **Hito 0 + Hito 1**
* al terminar el **Hito 2**, se ejecutan regresiones de **Hito 0 + Hito 1 + Hito 2**
* …
* al terminar el **Hito 22**, se ejecutan regresiones acumuladas de **Hito 0 a Hito 22**

La idea es que **ningún hito nuevo da por buena su implementación sin volver a demostrar que no ha roto nada ya consolidado**, lo cual es totalmente coherente con la prioridad que el documento da a la robustez de validación, la trazabilidad, la fluidez y la estabilidad del core.

## Observación importante para agentes IA

Aunque el roadmap está muy detallado, el propio documento insiste en que **no es un corsé absoluto**, sino una guía muy aproximada pero seria. Eso significa que un agente IA que implemente estos hitos debe respetar la dirección arquitectónica y los principios rectores, pero puede ajustar detalles concretos si la implementación real, la validación o las pruebas lo aconsejan. Lo importante no es clavar al milímetro cada nombre interno, sino preservar estas propiedades: separación de responsabilidades, validación fuerte, persistencia como fuente de verdad, integración formal con ytdl-sub, caché y cola como partes centrales del comportamiento operativo, API propia y UX aplazada hasta que el núcleo esté terminado y probado.
