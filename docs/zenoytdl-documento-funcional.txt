Documento funcional de alto nivel

Zenoytdl

Propuesta funcional, técnica y arquitectónica preliminar



1. Naturaleza de este documento

El presente documento define una propuesta funcional, técnica y arquitectónica de alto nivel para la herramienta Zenoytdl.

Su objetivo es servir como marco de referencia aproximado para el diseño e implementación del sistema. No debe interpretarse como una especificación cerrada, inmutable o absolutamente innegociable. Al contrario, debe entenderse como una idea muy cercana a lo que se quiere construir, con suficiente nivel de detalle como para orientar decisiones de arquitectura, diseño de configuración, modelado de datos, validaciones, integración con la capa inferior y evolución futura del proyecto.

En consecuencia:

algunas decisiones aquí descritas deben considerarse preferentes, pero no irrevocables,

ciertas estructuras pueden reajustarse durante la implementación,

y algunos nombres, claves, ficheros o responsabilidades podrán afinarse si el diseño final lo requiere.

Aun así, el documento pretende dejar lo más claro posible qué se quiere lograr, cómo debería organizarse, qué responsabilidades tiene cada parte, y cuáles son las prioridades reales del sistema.



2. Definición general de Zenoytdl

2.1 Qué es Zenoytdl

Zenoytdl es una herramienta de más alto nivel construida por encima de ytdl-sub, que utilizará a esta última como motor subyacente de resolución, descarga y parte del procesamiento, mientras ofrece una capa superior propia orientada a:

simplificar la complejidad de uso,

mejorar radicalmente la experiencia del usuario,

centralizar y normalizar la configuración,

abstraer las particularidades internas de ytdl-sub,

añadir semántica propia orientada a negocio y caso de uso real,

incorporar capacidades nuevas no resueltas directamente por la herramienta inferior,

exponer una interfaz programática consistente,

y sentar una base robusta para interfaces futuras mucho más amigables.

Zenoytdl no pretende reemplazar ytdl-sub, sino encapsularlo y aprovecharlo como base técnica, añadiendo una capa de orquestación, modelado, validación, persistencia, control y usabilidad superior.



2.2 Propósito principal

El propósito de Zenoytdl es convertir una base técnica potente pero compleja en un sistema:

más predecible,

más mantenible,

más reutilizable,

más extensible,

y considerablemente más amigable de operar.

Para ello, Zenoytdl debe permitir que el usuario trabaje con conceptos de más alto nivel, como por ejemplo:

perfiles,

suscripciones,

postprocesados,

overrides,

políticas de retención,

caché,

colas,

estado,

sincronización,

compilación,

conectividad con ytdl-sub,

y control de ejecución.

En lugar de obligar al usuario a modelar toda la lógica directamente con la semántica nativa de ytdl-sub, Zenoytdl debe ofrecer una semántica propia, más clara y más cercana a cómo el usuario piensa el problema real.



3. Objetivos funcionales del sistema

Zenoytdl debe perseguir, como mínimo, los siguientes objetivos funcionales:

3.1 Simplificación de configuración

La herramienta debe permitir definir comportamientos complejos mediante una configuración declarativa más clara que la de la capa inferior.

3.2 Abstracción semántica

El usuario debe poder trabajar con conceptos funcionales de alto nivel sin necesidad de conocer en profundidad el modelo interno de ytdl-sub.

3.3 Reutilización

Debe evitarse la repetición constante de reglas, parámetros y pipelines mediante la existencia de perfiles reutilizables.

3.4 Control fino

Las suscripciones deben poder heredar valores de perfiles, pero también poder alterarlos mediante overrides controlados.

3.5 Consistencia del modelo

Deben poder bloquearse configuraciones para evitar overrides ilegítimos o incoherencias semánticas.

3.6 Persistencia y trazabilidad

La herramienta debe recordar qué ha hecho, qué descargó, qué purgó, qué descartó y por qué.

3.7 Integración fiable con ytdl-sub

La comunicación entre ambas herramientas debe estar modelada de forma clara, estable y controlada.

3.8 Fluidez operativa

El sistema debe ser fluido incluso con grandes volúmenes de trabajo, gracias a un diseño explícito de colas, caché, validación eficiente y control de ejecución.

3.9 Exposición programática

Zenoytdl debe disponer de una API propia para poder ser operado externamente sin depender solo de ficheros YAML o interacción local.

3.10 Base sólida antes que interfaz visual

La UX será crítica, pero debe llegar solo después de que el núcleo esté terminado, validado y pasando todos los tests.



4. Principios rectores del diseño

El diseño de Zenoytdl debería basarse en los siguientes principios:

4.1 ytdl-sub como motor, no como interfaz de usuario

Zenoytdl debe usar ytdl-sub como backend técnico, pero no debe trasladar su complejidad directamente al usuario.

4.2 Capa de alto nivel real

Zenoytdl debe ser una capa con semántica propia, no un simple wrapper superficial.

4.3 Configuración declarativa, clara y legible

Los YAMLs deben ser entendibles, coherentes, mantenibles y orientados al dominio.

4.4 Separación estricta de responsabilidades

Cada fichero, módulo y componente debe tener una responsabilidad clara.

4.5 Validación fuerte antes de ejecutar

La mayor cantidad posible de errores debe detectarse en fases de validación y compilación, no durante la ejecución tardía.

4.6 Override con control explícito

La flexibilidad no debe romper la coherencia del sistema.

4.7 Persistencia como fuente de verdad operativa

La herramienta debe basarse en un estado persistido fiable, no en deducciones frágiles sobre el filesystem.

4.8 Rendimiento y fluidez como propiedad de diseño

La fluidez no debe dejarse para el final; debe estar contemplada desde la arquitectura mediante cola, caché, deduplicación y control de concurrencia.

4.9 UX tardía pero prioritaria

La UX no debe adelantarse al core, pero una vez que el núcleo esté estable, pasará a ser una prioridad máxima.



5. Alcance conceptual del sistema

Zenoytdl debería quedar organizado alrededor de los siguientes bloques funcionales principales:

configuración general,

perfiles,

suscripciones,

postprocesados,

sistema de overrides,

integración y compilación hacia ytdl-sub,

fichero de conectividad específico ytdl-sub-conf,

base de datos de estado,

sistema de caché,

gestor de colas,

API propia,

y futura capa UX desacoplada.



6. Modelo funcional de alto nivel

6.1 Configuración general

Agrupa la configuración técnica transversal de toda la herramienta.

6.2 Perfiles

Definen plantillas de comportamiento reutilizable.

6.3 Suscripciones

Representan fuentes concretas de contenido.

6.4 Postprocesados

Definen transformaciones o enriquecimientos aplicables al contenido.

6.5 Overrides

Permiten alterar parcialmente la herencia de un perfil, siempre bajo reglas de control.

6.6 Integración con ytdl-sub

Traduce el modelo interno de Zenoytdl hacia la capa inferior real.

6.7 Estado

Permite trazabilidad, persistencia operativa y prevención de redescargas indebidas.

6.8 Caché

Reduce trabajo repetitivo, acelera operaciones frecuentes y mejora fluidez.

6.9 Colas

Ordenan, priorizan y controlan la ejecución de trabajos de forma robusta y escalable.

6.10 API

Expone el sistema a automatización y futuras interfaces.

6.11 UX futura

Se desarrollará después, desacoplada del core, probablemente en un proyecto aparte.



7. Estructura mínima de configuración propuesta

Se propone una estructura basada, como mínimo, en los siguientes ficheros YAML obligatorios:

general.yaml

profiles.yaml

subscriptions.yaml

ytdl-sub-conf.yaml

Y opcionalmente, si durante la implementación se considera conveniente, podrían existir ficheros adicionales como:

postprocessing.yaml

plugins.yaml

defaults.yaml

locks.yaml

queue.yaml

cache.yaml

api.yaml

No obstante, como base mínima obligatoria, los cuatro ficheros anteriores deberían considerarse el núcleo de configuración.



8. Fichero general.yaml

8.1 Propósito

Contener toda la configuración técnica transversal de la herramienta que:

aplica a todo el sistema,

no pertenece a un perfil concreto,

no debe repetirse por suscripción,

y define el entorno operativo general.

8.2 Contenido esperado

Aquí deben ir, entre otros:

rutas o mecanismos de resolución de binarios,

directorios globales,

base de datos de estado,

ajustes técnicos de red,

histórico de descargas,

flags globales de ejecución,

parámetros generales de logging,

y configuraciones técnicas no relacionadas con la semántica de negocio.

8.3 Ejemplos de campos

ffmpeg_path

ffprobe_path

ytdl_sub_path

download_archive

global_output_directory

state_db_path

native_ytdl_sub_config

network_settings

binary_resolution

logging

temp_directory



9. Fichero profiles.yaml

9.1 Propósito

Definir los perfiles reutilizables que describen el comportamiento base de distintas clases de contenido.

9.2 Qué representa un perfil

Un perfil representa una plantilla operativa, es decir, una definición reutilizable que especifica:

el tipo semántico del perfil,

el pipeline base esperado,

el tipo de salida,

el formato final,

la calidad,

las políticas de retención,

los postprocesados por defecto,

y, en general, el comportamiento por defecto que heredarán las suscripciones asignadas a ese perfil.

9.3 Idea funcional

El perfil responde a la pregunta:

“¿Cómo debe comportarse este tipo de contenido por defecto?”



10. Fichero subscriptions.yaml

10.1 Propósito

Definir las fuentes concretas que el sistema debe seguir, inspeccionar, descargar o mantener sincronizadas.

10.2 Qué representa una suscripción

Una suscripción representa un origen real, por ejemplo:

la sección de vídeos de un canal,

la sección de shorts,

una playlist,

un vídeo suelto,

una fuente externa compatible.

10.3 Responsabilidades de una suscripción

Cada suscripción debe definir, como mínimo:

una identidad estable,

el tipo de elemento,

la fuente concreta,

el perfil asignado,

si está habilitada o no,

y cualquier override permitido.



11. Nuevo fichero obligatorio: ytdl-sub-conf.yaml

11.1 Propósito general

Este fichero debe existir como pieza obligatoria separada y especializada para modelar toda la información relevante a la conectividad, traducción, compatibilidad y comunicación perfecta entre Zenoytdl y ytdl-sub.

No debe confundirse con la configuración general del sistema. Aunque parte de su contenido tenga naturaleza técnica, su función no es definir el entorno global de Zenoytdl, sino actuar como contrato técnico de acoplamiento controlado entre ambas aplicaciones.

11.2 Motivación

La integración entre Zenoytdl y ytdl-sub no debe quedar repartida caóticamente entre múltiples ficheros ni escondida como lógica implícita difícil de rastrear.
Debe existir un fichero específico donde se concentre todo lo relacionado con:

compatibilidad de campos,

reglas de traducción,

presets puente,

mapeos de claves,

comportamiento de ejecución,

estrategias de compilación,

controles de validación cruzada,

restricciones derivadas de la capa inferior,

y particularidades necesarias para que el acoplamiento sea 100% fiable.

11.3 Qué debe contener

Este fichero debería modelar, entre otros, los siguientes aspectos:

11.3.1 Resolución de presets base y presets puente

Debe definir cómo un perfil de Zenoytdl se traduce a presets concretos de ytdl-sub.

11.3.2 Mapeo de claves

Debe especificar qué claves de Zenoytdl se traducen a qué claves, bloques o estructuras de ytdl-sub.

11.3.3 Reglas de normalización

Debe contener reglas para transformar valores antes de entregarlos a la capa inferior.

11.3.4 Compatibilidades e incompatibilidades

Debe reflejar qué combinaciones son válidas y cuáles no lo son.

11.3.5 Reglas de compilación

Debe definir cómo se construyen los artefactos efectivos que consumirá ytdl-sub.

11.3.6 Parámetros de invocación

Debe centralizar flags, argumentos, orden de construcción y mecanismos de llamada necesarios para una integración fiable.

11.3.7 Estrategias de fallback

Debe permitir definir cómo reaccionar cuando un valor de alto nivel no tenga traducción directa.

11.3.8 Validación cruzada

Debe incluir la lógica declarativa necesaria para comprobar que el modelo Zenoytdl es compatible con el modelo ytdl-sub.

11.3.9 Versionado de integración

Debe permitir saber con qué versión o rango de comportamiento esperado de ytdl-sub está alineado el proyecto.

11.4 Idea clave

ytdl-sub-conf.yaml debe ser, en esencia, el punto de verdad del contrato de acoplamiento entre Zenoytdl y ytdl-sub.



12. Perfiles previstos

Se contemplan inicialmente siete perfiles principales:

12.1 podcast

Perfil orientado a contenido largo que se quiere consumir como audio, como episodios, tertulias, entrevistas o programas completos.

12.2 tv

Perfil orientado a contenido vídeo largo o continuo, típicamente programas, series, secciones estables o formatos audiovisuales más extensos.

12.3 videos_sueltos

Perfil orientado a vídeos independientes o pequeños conjuntos agrupados por tema o biblioteca.

12.4 ambience_video

Perfil orientado a contenido visual de ambientación, normalmente largo o repetitivo, con necesidad frecuente de límite de duración.

12.5 ambience_audio

Perfil equivalente al anterior pero centrado en audio de ambientación o escucha continua.

12.6 music_playlist

Perfil orientado a playlists musicales o colecciones de audio con foco en organización y biblioteca sonora.

12.7 shorts

Perfil orientado a contenido corto con pipeline adaptado a piezas breves.

Estos perfiles no deben entenderse necesariamente como una lista absolutamente cerrada o imposible de ampliar, pero sí como el conjunto base actualmente previsto.

17. Sistema de overrides

17.1 Propósito

Permitir que una suscripción, o eventualmente un elemento concreto dentro de ella, pueda sobrescribir parcialmente valores heredados del perfil.

17.2 Necesidad de control

No todos los campos deben ser libremente sobrescribibles.

17.3 Política recomendada

Cada campo susceptible de override debería poder marcarse al menos con uno de estos estados:

allowed

restricted

forbidden

O una semántica equivalente.

17.4 Ejemplo práctico

Un perfil podcast puede imponer que:

el tipo final es audio,

la extensión final es mp3,

el preset base es de audio,

y bloquear por completo el override de esos campos.

17.5 Resultado

Se gana flexibilidad sin destruir la consistencia semántica del perfil.



18. Control de retención, purga y limpieza

18.1 Significado de max_items

Cuando un perfil o suscripción define max_items = X, eso significa que solo deben conservarse físicamente X elementos.

18.2 Comportamiento básico

si hay menos de X, se sigue descargando contenido nuevo,

si ya hay X y aparecen elementos nuevos, se purgan los más antiguos,

el objetivo es conservar siempre la ventana más reciente de contenidos.

18.3 Criterio de antigüedad

El criterio prioritario debe ser la fecha de publicación, no simplemente la fecha de descarga.

18.4 Necesidad de trazabilidad

El sistema debe registrar qué fue purgado y por qué.



19. Persistencia y base de datos de estado

Zenoytdl debe apoyarse en una base de datos SQLite como mínimo para almacenar estado operativo.

19.1 Objetivos de la persistencia

evitar redescargas infinitas,

mantener trazabilidad,

registrar purgas,

registrar descartes,

registrar sincronizaciones,

soportar colas y caché,

y habilitar consultas desde la API.

19.2 Información recomendable a persistir

identificador estable del elemento,

identificador de suscripción,

perfil aplicado,

timestamps relevantes,

estado de descarga,

estado de purga,

motivo de descarte,

hash o firma de configuración efectiva,

estado de postprocesado,

referencias a caché,

métricas de ejecución.



20. API propia de Zenoytdl

20.1 Necesidad de API

Zenoytdl debe exponer una API propia para ser operado programáticamente.

20.2 Motivos

automatización externa,

consumo por futuras interfaces,

desacoplamiento del core,

trazabilidad de operaciones,

control remoto o local estructurado,

integración con otros sistemas.

20.3 Capacidades esperables

La API debería poder exponer operaciones como:

listar perfiles,

obtener un perfil,

validar configuración,

listar suscripciones,

crear suscripción,

actualizar suscripción,

resolver configuración efectiva,

consultar cola,

ejecutar sincronización,

consultar historial,

consultar caché,

reintentar trabajos fallidos,

inspeccionar elementos descargados o purgados.

20.4 Papel arquitectónico

La API debe ser una frontera estable entre el core y las futuras interfaces.



21. Sistema de caché

Aquí se introduce una ampliación importante que no estaba desarrollada antes y que es muy recomendable para que el sistema sea realmente fluido.

21.1 Objetivo del sistema de caché

Zenoytdl debe disponer de un sistema de caché potente que reduzca trabajo repetitivo, minimice operaciones innecesarias y mejore significativamente la sensación de fluidez del sistema, especialmente cuando existan muchas suscripciones, múltiples validaciones, compilaciones frecuentes y ejecuciones reiteradas.

21.2 Qué problemas resuelve

La caché debe servir para evitar:

revalidaciones idénticas innecesarias,

recomputación constante de configuraciones efectivas,

recompilación reiterada hacia ytdl-sub,

consultas repetidas a metadatos ya resueltos,

reevaluación continua de estados recientes,

relectura excesiva de fuentes cuando no ha cambiado nada relevante,

y sobrecoste innecesario en operaciones de orquestación.

21.3 Tipos de caché recomendados

21.3.1 Caché de configuración compilada

Debe almacenar la versión compilada efectiva de:

perfil,

suscripción,

overrides,

traducción a ytdl-sub,

firma resultante.

Esto permite no recompilar cada vez una misma configuración si nada ha cambiado.

21.3.2 Caché de resolución de metadatos

Debe almacenar resultados de inspección o resolución previa sobre elementos y fuentes, con TTL o invalidación por firma.

21.3.3 Caché de validación

Debe permitir reutilizar resultados de validaciones ya realizadas sobre configuraciones equivalentes.

21.3.4 Caché de mapeo y traducción

Debe guardar resultados del proceso de traducción entre el modelo de Zenoytdl y el modelo efectivo de ytdl-sub.

21.3.5 Caché operativa de estado reciente

Debe permitir conocer rápidamente si una suscripción o un trabajo ha sido procesado hace muy poco y evitar reevaluaciones excesivas en ventanas cortas.

21.4 Estrategias de invalidación

Toda caché útil necesita reglas claras de invalidación. Se recomienda soportar, como mínimo:

invalidación por cambio de fichero,

invalidación por cambio de hash de configuración,

invalidación por cambio de versión del contrato ytdl-sub-conf,

invalidación por TTL,

invalidación manual,

invalidación por error o inconsistencia detectada.

21.5 Requisitos funcionales de la caché

El sistema de caché debería:

ser transparente para el usuario cuando sea posible,

ser inspeccionable desde la API,

permitir limpieza parcial o total,

soportar estadísticas de hit/miss,

no comprometer la consistencia,

y priorizar siempre corrección por encima de velocidad.

21.6 Relación con UX y fluidez

Una caché bien diseñada será esencial para que la TUI futura y la GUI web puedan ofrecer una sensación de rapidez, navegación suave y respuesta inmediata.



22. Gestor de colas

También se añade aquí un componente clave para garantizar una experiencia fluida y una ejecución robusta.

22.1 Necesidad del gestor de colas

Zenoytdl debe incorporar un sistema de colas potente y bien estructurado para organizar, priorizar, distribuir y controlar todas las tareas que realice el sistema.

No conviene depender de ejecuciones lineales poco controladas o de simples bucles monolíticos. La herramienta debe tener un modelo de trabajo explícito.

22.2 Qué debe gestionar la cola

El gestor de colas debería poder manejar, como mínimo, trabajos de:

validación de configuración,

compilación hacia ytdl-sub,

inspección de suscripciones,

sincronización,

descarga,

postprocesado,

purga,

actualización de estado,

limpieza de caché,

reintentos,

mantenimiento técnico.

22.3 Objetivos del sistema de colas

mantener fluidez,

evitar bloqueos innecesarios,

ordenar correctamente el trabajo,

permitir concurrencia controlada,

evitar saturación,

facilitar reintentos,

soportar priorización,

y desacoplar bien las fases del pipeline.

22.4 Modelo recomendado

Se recomienda un modelo de colas con varias clases de trabajo, por ejemplo:

cola de validación

cola de compilación

cola de sincronización

cola de descarga

cola de postprocesado

cola de mantenimiento

No necesariamente deben ser colas totalmente aisladas físicamente, pero sí deberían existir al menos como clases lógicas diferenciadas.

22.5 Prioridades

Los trabajos deberían poder tener prioridad, por ejemplo:

alta,

normal,

baja,

mantenimiento.

Esto permitirá que acciones críticas o interactivas no queden bloqueadas detrás de tareas pesadas de fondo.

22.6 Estados del trabajo

Cada trabajo debería tener estados trazables, tales como:

queued

scheduled

running

waiting

completed

failed

retry_pending

cancelled

dead_letter

22.7 Reintentos

El sistema de colas debería soportar reintentos configurables con:

número máximo de reintentos,

backoff progresivo,

clasificación de errores recuperables y no recuperables,

y paso a cola muerta o equivalente cuando un trabajo no pueda completarse.

22.8 Deduplicación

El gestor de colas debería poder detectar trabajos duplicados o equivalentes para evitar:

reevaluaciones idénticas,

compilaciones duplicadas,

sincronizaciones concurrentes de la misma suscripción,

o postprocesados repetidos sobre el mismo artefacto.

22.9 Concurrencia controlada

No toda tarea debe correr en paralelo sin límite.
Debería existir control fino de concurrencia por:

tipo de cola,

suscripción,

perfil,

origen,

recursos del sistema,

o clase de operación.

22.10 Integración con caché y estado

El sistema de colas no debe ser una pieza aislada. Debe integrarse con:

la base de datos de estado,

el sistema de caché,

la API,

y el módulo de ejecución.

22.11 Beneficio sobre la fluidez

Un buen gestor de colas es uno de los factores más importantes para que el sistema se perciba como rápido, ordenado, robusto y profesional.



23. Relación entre caché, colas y fluidez

La combinación de caché y gestor de colas no debe verse como un añadido accesorio, sino como parte central del comportamiento operativo de Zenoytdl.

23.1 La caché aporta

menos recomputación,

menor latencia,

menor carga repetitiva,

respuestas más rápidas.

23.2 La cola aporta

orden,

control,

priorización,

aislamiento de trabajo,

mejor aprovechamiento de recursos.

23.3 Juntas permiten

interacción más ágil,

menor sensación de bloqueo,

mejor respuesta ante grandes cargas,

preparación ideal para TUI y GUI futuras,

y una arquitectura más madura.



24. UX: prioridad máxima, pero en el momento correcto

Este punto debe quedar muy claro.

24.1 La UX es ultra importante

La UX de Zenoytdl será extraordinariamente importante. El objetivo final es que la herramienta resulte altamente amigable de usar, incluso si internamente su arquitectura es compleja.

24.2 Pero no debe abordarse antes de tiempo

La UX no debe convertirse en prioridad práctica antes de que el sistema base esté realmente terminado.

24.3 Condición obligatoria previa

Solo cuando:

el core esté completo,

la integración con ytdl-sub esté consolidada,

la validación sea robusta,

las colas funcionen correctamente,

la caché esté integrada,

la persistencia esté estabilizada,

la API sea fiable,

y todos los tests previstos estén pasando,

entonces la UX pasará a ser prioridad máxima.

24.4 Interfaces previstas

La UX futura deberá materializarse, como mínimo, en:

un panel de control TUI,

y una GUI web muy refinada.

24.5 Proyecto aparte

Tanto la TUI como la GUI deberían realizarse en un proyecto separado, desacoplado del core principal, para evitar contaminar la base funcional con decisiones prematuras de interfaz.



25. Orden recomendado de prioridades

El orden lógico y recomendable de implementación sería aproximadamente este:

modelado de dominio,

diseño de configuración,

validación,

compilación e integración con ytdl-sub,

persistencia,

control de retención y purga,

sistema de caché,

gestor de colas,

API,

test suite completa,

estabilización final,

UX avanzada en proyecto aparte.

Este orden tampoco debe entenderse como matemáticamente rígido, pero sí como la secuencia más sensata para no construir una interfaz vistosa sobre una base aún inmadura.



26. Reglas prácticas de separación por niveles

26.1 Perfil

Define cómo debe comportarse el procesamiento.

26.2 Suscripción

Define qué fuente concreta se sigue.

26.3 Postprocesado

Define qué transformaciones se aplican.

26.4 Configuración general

Define el entorno técnico común.

26.5 ytdl-sub-conf

Define el contrato técnico de integración con la capa inferior.

26.6 Caché

Define cómo se evitan recomputaciones innecesarias.

26.7 Cola

Define cómo se organiza y controla el trabajo.

26.8 Estado

Define la memoria operativa del sistema.

26.9 API

Define cómo puede interactuarse programáticamente con el sistema.

26.10 UX futura

Define cómo el usuario operará la herramienta de forma altamente amigable cuando todo lo demás esté listo.

  integration_version: "1"

  compatibility:
    supported_ytdl_sub_versions:
      - ">=0.5.0"
    strict_mode: true

  preset_mapping:
    podcast: "base::audio"
    tv: "base::video"
    videos_sueltos: "base::video"
    ambience_video: "base::video"
    ambience_audio: "base::audio"
    music_playlist: "base::audio"
    shorts: "base::video"

  field_mapping:
    profile_type: "internal_only"
    output_directory: "output_options.output_directory"
    format.type: "preset_family"
    format.extension: "output_options.file_extension"
    quality: "download_options.quality"
    max_items: "retention.max_items"
    min_duration: "filters.min_duration"
    library_name: "metadata.library_name"

  translation_rules:
    - when:
        profile_type: "podcast"
      set:
        format.type: "audio"
        preset_family: "audio"

    - when:
        profile_type: "tv"
      set:
        format.type: "video"
        preset_family: "video"

  invocation:
    mode: "subprocess"
    pass_compiled_yaml_via: "file"
    compiled_output_directory: "/data/compiled-ytdl-sub"
    clean_previous_compiled_files: true

  fallback_policy:
    on_unknown_field: "error"
    on_unsupported_mapping: "error"
    on_partial_translation: "warn_and_abort"

  validation:
    require_mapped_profile: true
    require_supported_extension: true
    require_preset_mapping: true
    reject_incompatible_audio_video_mix: true

29. Consideraciones de implementación

29.1 Documento no cerrado

Nada de lo aquí descrito debe interpretarse como un corsé absoluto.
El proyecto podrá y deberá reajustar detalles cuando la implementación real, la validación o las pruebas lo aconsejen.

29.2 Pero sí es una guía muy aproximada

Aun así, este documento representa una aproximación muy cercana a la herramienta que se desea construir y por tanto debe usarse como referencia seria para diseño y desarrollo.

29.3 Prioridad de calidad

Debe priorizarse:

claridad de arquitectura,

coherencia del modelo,

robustez de validación,

trazabilidad,

fluidez,

y estabilidad del core.



30. Conclusión funcional consolidada

Zenoytdl debe concebirse como una capa superior, robusta y bien modelada sobre ytdl-sub, cuyo valor principal no está únicamente en descargar contenido, sino en ofrecer una semántica más clara, una configuración reutilizable, una integración fiable, una persistencia coherente, una arquitectura preparada para automatización y un futuro entorno de uso altamente amigable.

La herramienta debe estructurarse alrededor de:

configuración general,

perfiles,

suscripciones,

postprocesados,

overrides,

estado persistente,

integración formal con ytdl-sub mediante ytdl-sub-conf.yaml,

un sistema de caché potente,

un gestor de colas sólido,

una API propia,

y una futura TUI + GUI web desacopladas del núcleo.

La UX será críticamente importante, pero solo deberá abordarse cuando el sistema base esté terminado, estabilizado y pasando todos los tests previstos. En ese punto, el objetivo será lograr una herramienta altamente amigable y refinada, sin sacrificar la solidez del núcleo técnico sobre el que se apoya.
